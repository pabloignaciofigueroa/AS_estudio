# Workflow Web Exportable

Este archivo esta pensado para ser entendido por:

- humanos
- ChatGPT / Codex
- Claude Code
- otros agentes o sistemas de automatizacion

Su funcion es servir como protocolo base para proyectos futuros de paginas web comerciales o de venta.

## 1. Principio central

No empezar por codigo.

El orden correcto es:

1. entender el negocio
2. definir el workflow
3. reunir referencias
4. reunir marca
5. reunir activos reales
6. definir alcance
7. estructurar
8. redactar
9. diseñar
10. construir
11. iterar

## 2. Regla clave

### Fase 1: workflow

Siempre preguntar.

### Fase 2: construccion

Preguntar menos y ejecutar mas.

## 3. Inputs minimos obligatorios

Antes de construir, reunir esta informacion:

### Negocio

- rubro
- objetivo del sitio
- conversion esperada
- que se quiere vender
- que no se quiere mostrar

### Referencia

- 1 referente principal
- 1 o 2 referentes secundarios, si existen
- explicacion de que se quiere copiar del referente

### Marca

- nombre exacto
- logotipo
- isotipo
- imagotipo
- paleta
- tipografias, si existen
- tono de marca

### Activos

- imagenes
- videos
- renders
- proyectos
- testimonios
- textos reales

### CTA

- a donde debe llevar cada boton
- canal principal:
  - web
  - WhatsApp
  - email
  - Instagram
  - formulario

## 4. Checklist de preguntas para fase workflow

Estas preguntas deben responderse antes de diseñar:

1. Cual es el objetivo exacto del sitio?
2. Es una preview, una landing, un portafolio o una web completa?
3. Cual es el rubro?
4. Cual es el referente principal?
5. Que parte del referente quieres imitar?
6. Cual es la marca y su jerarquia visual?
7. Que archivos de identidad existen?
8. Que imagenes reales existen hoy?
9. Que contenido real falta?
10. Que podemos inventar provisionalmente?
11. Cual es la accion principal que debe tomar la persona?
12. Cual es el tono:
    - editorial
    - corporativo
    - comercial
    - tecnico
    - premium

## 5. Fase de definicion

Con las respuestas, producir estos entregables:

### Entregable A: direcciones del proyecto

- objetivo
- tono
- referencia
- alcance
- CTA

### Entregable B: blueprint del referente

- estructura
- patrones visuales
- tono
- navegacion
- jerarquia
- elementos a copiar
- elementos a evitar

### Entregable C: workflow del proyecto

- como se trabajara
- quien define que
- como se versiona
- cuando se pregunta
- cuando se ejecuta

## 6. Fase de estructura

Antes de diseñar, definir:

- sitemap o estructura
- wireframe textual
- secciones
- orden de lectura
- bloques principales
- CTAs

## 7. Fase de copy

Reglas:

- si no hay texto real, usar copy provisional
- marcar siempre lo provisional
- escribir con el tono aprobado

Nunca:

- inflar el lenguaje
- prometer de mas
- inventar datos sin marcarlo

## 8. Fase de diseño

Orden correcto:

1. sistema base
2. header
3. hero
4. secciones centrales
5. CTAs
6. footer
7. estados hover
8. responsive

## 9. Fase de construccion

En construccion:

- ejecutar en archivos reales
- usar versiones cuando el cambio sea importante
- no destruir versiones previas
- nombrar versiones con claridad

### Regla de versionado

Si el proyecto nace como `index.html`, luego:

- `indexv2.html`
- `indexv3.html`
- `indexv4.html`

y sus equivalentes:

- `indexv2.css`
- `indexv2.js`

Nunca mezclar cambios grandes en una sola version viva sin respaldo.

## 10. Fase de iteracion

Cada iteracion debe responder:

- que se cambia
- por que se cambia
- en que version vive
- que problema resuelve

### Tipos de cambio

- estructural
- visual
- contenido
- interaccion
- responsive
- conversion

## 11. Reglas de colaboracion humano + agente

### El usuario define

- negocio
- criterio
- gusto
- correccion
- aprobacion final

### El agente define

- traduccion a decisiones concretas
- estructura
- copy provisional
- diseño
- codigo
- orden de implementacion

## 12. Reglas de comunicacion

Pedir cambios con esta formula:

`componente + cambio + intencion`

Ejemplos:

- "header, hazlo mas sobrio"
- "quita ese boton, no aporta"
- "iguala el spacing del bloque inferior con el superior"
- "no reemplaces, elimina"

## 13. Señales de que falta preguntar mas

Si ocurre alguna de estas cosas, volver a workflow:

- demasiadas vueltas sin cerrar
- cambios contradictorios
- uso dudoso de marca
- dudas sobre el objetivo comercial
- no esta claro que convierte
- no esta claro que mostrar y que esconder

## 14. Cuando pasar a otro sistema

Cambiar de sistema solo cuando:

- el problema necesita depuracion visual en vivo
- hay una falla tecnica dificil de aislar
- se necesita inspector de navegador
- se necesitan varias comparaciones visuales rapidas
- ya hubo 3 o mas rondas sobre el mismo micro-ajuste

## 15. Prompt base exportable para cualquier agente

Usar este bloque al iniciar un proyecto nuevo:

```text
Proyecto: [nombre del proyecto]
Rubro: [rubro]
Objetivo: [que debe lograr la pagina]
Tipo de sitio: [preview / landing / portafolio / sitio completo]
Referencia principal: [url o archivo]
Que se quiere tomar del referente: [estructura / tono / look / ritmo / CTA]
Marca: [nombre]
Activos de marca: [logotipo / isotipo / imagotipo / paleta / tipografia]
Activos de contenido: [imagenes / videos / proyectos / testimonios / textos]
CTA principal: [Instagram / WhatsApp / formulario / email / otro]
Tono deseado: [editorial / premium / corporativo / comercial / tecnico]
Restricciones: [lo que no se quiere]

Antes de construir:
1. resume el proyecto
2. haz preguntas de workflow
3. arma el workflow
4. propone estructura
5. recien despues construye
```

## 16. Orden operativo recomendado para proyectos futuros

1. Ir al rubro
2. Elegir referencia principal
3. Analizar referente
4. Revisar marca:
   - logo
   - isotipo
   - imagotipo
   - paleta
5. Revisar activos reales:
   - imagenes
   - proyectos
   - textos
6. Definir objetivo comercial
7. Armar workflow
8. Armar wireframe
9. Redactar copy
10. Diseñar
11. Construir
12. Versionar
13. Pulir

## 17. Regla final

El workflow no es burocracia.
Es el sistema que evita perder tiempo, repetir errores y diseñar sin rumbo.
